## 3) Diagnostic Judgment

高置信结论（7.3.3）：

1. 主要瓶颈不在 JVM GC 或堆内存，而在并发链路中的排队与争用。
2. 两个优先怀疑点：
- 网关吞吐瓶颈（运行方式为单 `uvicorn` worker，且转发层每请求新建 `httpx.AsyncClient`，连接复用弱）。
- order-service 到 MySQL 的连接池容量默认值偏小（`max=10`），峰值窗口出现明显等待。

## 4) Tuning Plan for Next Verification (7.3.4)

1. 网关并发模型调优（优先级 P1）
- 目标：缓解网关单进程瓶颈与排队。
- 建议：改为多 worker 运行（例如 `uvicorn --workers 2~4`，按主机核数试探）。

2. 网关转发连接复用（优先级 P1）
- 目标：降低每请求建连/销毁成本。
- 建议：将 `AsyncClient` 由“请求内创建”改为“进程级复用”，并配置连接池上限与 keepalive。

3. order-service 数据库连接池调优（优先级 P1）
- 目标：降低连接等待时间。
- 建议新增环境变量（A/B 验证）：
  - `SPRING_DATASOURCE_HIKARI_MAXIMUM_POOL_SIZE=20`
  - `SPRING_DATASOURCE_HIKARI_MINIMUM_IDLE=10`
  - `SPRING_DATASOURCE_HIKARI_CONNECTION_TIMEOUT=3000`

4. JVM 基线参数显式化（优先级 P2）
- 目标：将运行时行为固定，便于后续可重复比较。
- 建议（示例）：`-Xms1g -Xmx1g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m -XX:+HeapDumpOnOutOfMemoryError`

5. JVM 原生命令采样能力补齐（优先级 P2）
- 目标：满足 `jstat/jstack/jmap -histo` 证据链。
- 现状：当前运行镜像无 JDK 工具（见 7.3.2 记录）。
- 建议：准备 diagnostics 变体镜像或 sidecar 工具容器后，在 7.3.4 复测中补采。
